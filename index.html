Adaptive Yard Routing

Yard management software is rarely useful when it stays at the altitude of “optimize operations.” To understand adaptive routing, we must descend from the dashboard and experience the yard as a living, breathing state machine defined by queues, handoffs, and relentless contention under time pressure.

The failure of most logistics automation begins at the gate. At 14:00 on a Tuesday, an inbound carrier arrives. The driver rolls down the window and hands the guard a bill of lading. Behind him, three more trucks sit idling, bleeding air-brakes on the access road. Inside the facility, the control room is staring at a clipboard, a radio, and a static spreadsheet that was obsolete the second it was printed.

The yard supervisor needs to assign a dock. But assigning a dock requires knowing which docks are empty, which docks are supposed to be empty but still have a lingering trailer, and which empty slots are reserved for an outbound load scheduled to depart in twenty minutes. If the supervisor guesses wrong, the inbound truck is routed to a full dock, where it blocks the fire lane. A jockey is then radioed to blindly search the yard for an empty drop lot. Suddenly, three disparate assets—a carrier, a jockey, and a dock—are deadlocked.

This is not an "optimization" problem. It is a state-management crisis. The yard is a massive, physical state machine where entities (trucks, trailers, jockeys, docks) change status in real time. Adaptive routing is the discipline of catching those state changes the split second they occur and automatically resolving the next routing instruction before a human has to pick up a radio.

[STATE_01]: Gate-In and Entity Registration

The moment the guard scans the inbound paperwork, the system does not merely log a timestamp. It instantiates a digital twin of the trailer. This entity carries properties: length, temperature constraints (reefer vs. dry van), cargo priority, and carrier identity. Before the physical gate arm rises, the decision engine evaluates the state of the yard against the inbound entity properties.

We are not relying on predictive AI here. We are relying on deterministic, highly constrained state transitions. The engine queries the dock matrix. It filters out docks that cannot accommodate a 53-foot reefer. It filters out docks assigned to conflicting carriers. It finds the nearest compatible, verifiably empty dock—or, if no dock is available, the nearest drop-lot slot. The driver receives a printed ticket with a barcode and a specific parking coordinate: SLOT_B4.

[STATE_02]: The Handoff and Request Generation

The truck parks at SLOT_B4. The driver unhooks the trailer and exits the facility. Now, the trailer exists in a dormant state: STAGED_DROPPED. The yard software knows it is sitting in B4, but a warehouse manager inside the building needs that trailer's cargo at DOCK_02 immediately.

Old logic required the warehouse manager to call the yard supervisor, who would then blast a request over the radio: "Hey, who's free to grab a reefer from B4 and put it on door 2?" This creates a race condition. Two jockeys might drive toward B4, wasting fuel and time, or no jockey responds because they assume someone else is handling it. The adaptive routing engine intercepts this request entirely.

When the warehouse manager clicks "Call to Dock" in their portal, the system triggers an asynchronous move request. The engine evaluates the live GPS positions of every jockey truck in the yard, checks their current task queue, and calculates the exact travel distance from their location to B4, then to Dock 02. It selects the optimal jockey and pushes a structured move task directly to their cabin tablet.

# Pseudo-logic for the jockey assignment state transition
def assign_move_request(trailer_id, target_dock):
    trailer = get_entity(trailer_id)
    trailer.state = "REQUESTED"
    
    # Filter jockeys by availability and capabilities (e.g., reefer hookups)
    available_jockeys = get_jockeys(state="IDLE")
    
    # Optimize assigning the nearest asset
    best_jockey = min(
        available_jockeys, 
        key=lambda j: calculate_distance(j.location, trailer.location)
    )
    
    # Dispatch the task to the selected terminal
    best_jockey.push_task({
        "action": "MOVE",
        "source": trailer.location,     # "ZONE_B4"
        "destination": target_dock,     # "DOCK_02"
        "entity": trailer.id
    })
    
    return best_jockey.id

[STATE_03]: Execution and Exception Handling

The jockey accepts the task on their screen with a single tap. The system transitions the trailer to IN_TRANSIT. The control room's dashboard updates instantly. But yards are chaotic, and software that assumes a perfect golden path will fail in production. What happens when the jockey arrives at B4 and the trailer has a flat tire?

In a manual yard, the jockey radios in, the supervisor forgets to write it down, and three hours later the warehouse is screaming about missing cargo. In the adaptive routing system, the jockey taps an "Exception" button on their tablet, selects "Maintenance Issue," and attaches a quick photo. The system immediately rolls the trailer state to MAINTENANCE_HOLD, alerting the shop. Crucially, the engine instantly kicks the move request back into the pool, finding the next available priority trailer to supply DOCK_02 without human intervention.

[STATE_04]: What Changes for the Operator

The control room stops being a center of frantic coordination. The yard supervisor transforms from a human router into an exception manager. Because the system autonomously matches nearest empty docks, continuously guides arriving trucks, and commands the jockey terminals, the supervisor spends zero time yelling into a radio about blind spots.

The metrics that matter—dwell time, move turnaround times, dock utilization—fall into line naturally. The automation strips away the latency between decisions. In logistics, time is crushed in the handoffs. By digitizing those handoffs and strictly enforcing state boundaries, the yard stops swallowing trailers whole. It becomes a predictable, observable engine.
